## Executive Summary

A comprehensive Quality Assurance audit was conducted on the DK Vroom application — a full-stack Next.js automotive marketplace covering 7 core modules (Rent, Buy & Sell, Repair/Workshop, Insurance, Auction, Loan, Continue Loan). The audit covered the complete codebase including 31 API routes, 78+ frontend components, Prisma ORM schema with 17 models, authentication/authorization systems, payment flow, and three role-based dashboards.

**Overall Assessment:** The application demonstrates a solid architectural foundation with well-structured code, comprehensive data models, and good separation of concerns. However, **22 bugs/defects were identified** (3 Critical, 7 High, 7 Medium, 5 Low), along with **5 security findings**, **3 performance issues**, and **4 accessibility concerns**.

**Critical Issues Detected:**
1. Admin notification for payment receipt upload sends to non-existent user ID `'admin'`
2. Payment QR code uses hardcoded production domain `https://dkvroom.com` — breaks on all other environments
3. Platform fee percentage inconsistency: 5% in Booking module vs 10% in Continue Loan module

**Production Readiness:** **CONDITIONAL PASS** — The core flows are functional but the critical and high-priority issues must be resolved before production deployment, particularly around the payment verification flow and security hardening.

## Bugs Identified

### Bug ID: BUG-001
**Severity:** 🔴 **Critical**
**Module:** Payment → Notifications
**Description:** When a customer uploads a payment receipt, the system sends a notification to userId `'admin'` (string literal). This user ID does not exist in the database, so no admin actually receives the notification.
**File:** `src/app/api/payments/[id]/route.ts` (line 132-137)
**Steps to Reproduce:**
1. Login as a customer
2. Create a booking
3. Go to payment page
4. Upload a receipt
5. Check the `notifications` table for a record with userId `'admin'`
**Expected Result:** Notification created for each admin user in the system
**Actual Result:** Notification created with userId = `'admin'` — a non-existent user ID — effectively lost
**Root Cause:** The code uses the literal string `'admin'` instead of querying for actual admin user IDs
**Recommended Fix:** Query for admin users and send notifications to each:
```typescript
const admins = await db.user.findMany({ where: { role: 'admin' } })
for (const admin of admins) {
  await db.notification.create({
    data: { userId: admin.id, title: 'Payment Receipt Uploaded', ... }
  })
}
```

### Bug ID: BUG-002
**Severity:** 🔴 **Critical**
**Module:** Payment → QR Code
**Description:** The payment QR code value is hardcoded to `https://dkvroom.com/pay?ref=...` regardless of the actual deployment environment. In development/staging, this will direct users to the production site.
**File:** `src/components/payment-page.tsx` (line 237)
**Steps to Reproduce:**
1. Run application on `localhost:3000`
2. Create a booking
3. Navigate to payment page
4. Inspect QR code value
**Expected Result:** QR code links to the current environment's URL
**Actual Result:** QR code always links to `https://dkvroom.com`
**Root Cause:** Hardcoded URL instead of using environment variable `NEXT_PUBLIC_APP_URL`
**Recommended Fix:**
```typescript
const appUrl = process.env.NEXT_PUBLIC_APP_URL || 'http://localhost:3000'
const qrValue = `${appUrl}/pay?ref=${booking.bookingId || 'DEMO'}&amount=${booking.amount}`
```

### Bug ID: BUG-003
**Severity:** 🔴 **Critical**
**Module:** Auth → Route Protection (Customer Dashboard)
**Description:** In `app-shell.tsx`, the customer dashboard redirect logic is inverted — it redirects customers AWAY from their own dashboard instead of allowing them to access it.
**File:** `src/components/app-shell.tsx` (lines 88-90)
**Steps to Reproduce:**
1. Login as a customer
2. Navigate to `/customer-dashboard`
3. Observe redirect to `/`
**Expected Result:** Customer sees their dashboard with bookings, payments, etc.
**Actual Result:** Customer is redirected to home page
**Root Cause:** The condition at line 88-90: `if (pathname?.startsWith('/customer-dashboard') && user.role === 'customer')` redirects — but this is the CORRECT role. It should redirect non-customers, not customers.
**Recommended Fix:**
```typescript
if (pathname?.startsWith('/customer-dashboard') && user.role !== 'customer') {
  router.replace('/')
}
```

### Bug ID: BUG-006
**Severity:** 🟠 **High**
**Module:** Security → Rate Limiting
**Description:** The rate limiting implementation uses an in-memory `Map<string, { count, resetTime }>` that resets on every server restart and doesn't scale across multiple server instances. In production with multiple pods/instances, each instance maintains its own count.
**File:** `src/lib/security/middleware.ts` (line 4)
**Expected Result:** Rate limiting should be persistent and shared across instances
**Actual Result:** In-memory only — resets on restart per-instance
**Recommended Fix:** Use external store (Redis, database, or even file-based) for production rate limiting:
```typescript
// Option: Use database-based rate limiting
const record = await db.rateLimit.upsert({ where: { key }, create: { key, count: 1, resetTime }, update: { count: { increment: 1 } } })
```

### Bug ID: BUG-007
**Severity:** 🟠 **High**
**Module:** Security → CSRF
**Description:** The application relies solely on same-origin policy and JWT Bearer tokens for CSRF protection. No anti-CSRF tokens are implemented. A malicious site could potentially make requests on behalf of an authenticated user.
**Files:** All `src/app/api/*/route.ts` files
**Expected Result:** Anti-CSRF token validation on state-changing requests
**Actual Result:** No CSRF protection mechanism beyond what the browser provides by default
**Recommended Fix:** Implement double-submit cookie pattern or CSRF tokens for all `POST/PUT/DELETE` requests:
```typescript
// In middleware or route handler
const csrfToken = request.headers.get('x-csrf-token')
// Validate against session-stored token
```

### Bug ID: BUG-008
**Severity:** 🟠 **High**
**Module:** Security → Token Storage
**Description:** The JWT token is stored in `localStorage` (`dkvroom_token` key). This makes it accessible to any JavaScript executing in the same origin, creating a persistent XSS attack vector. If an attacker achieves XSS, they can steal the token and impersonate any user.
**File:** `src/lib/api.ts` (line 28-32)
**Expected Result:** Token should be stored in an HttpOnly cookie (inaccessible to JavaScript)
**Actual Result:** Token stored in `localStorage` with key `dkvroom_token`
**Recommended Fix:** Move to HttpOnly cookie-based token storage:
```typescript
// Set cookie server-side on login
response.headers.set('Set-Cookie', `token=${jwt}; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=604800`)
```

### Bug ID: BUG-009
**Severity:** 🟠 **High**
**Module:** Security → File Upload Validation
**Description:** File upload validation only checks the `Content-Type` header and file extension, both of which can be easily spoofed. A malicious user could upload a `.jpg` file containing JavaScript/PHP code, and if the server serves it with the wrong content-type, it could execute.
**Files:**
- `src/lib/file-upload.ts` (lines 26-33)
- `src/app/api/upload/route.ts` (lines 63-72)
**Expected Result:** File content validation to verify actual file type
**Actual Result:** Only MIME type header and file extension checked
**Recommended Fix:** Use `file-type` package or `sharp` to validate actual file content magic bytes:
```typescript
import { fileTypeFromBuffer } from 'file-type'
const buffer = await file.arrayBuffer()
const type = await fileTypeFromBuffer(buffer)
if (!type || !allowedMimes.includes(type.mime)) throw new Error('Invalid file type')
```

### Bug ID: BUG-013
**Severity:** 🟡 **Medium**
**Module:** OTP/Rate Limiting
**Description:** OTP sending and verification endpoints (`/api/auth/send-otp`, `/api/auth/verify-otp`) are only protected by the auth rate limiter (10/15min per IP). There is no per-email rate limiting, meaning an attacker could brute-force OTP codes or spam a single email address.

**Actual Result:** Only IP-based rate limiting applied

```typescript
const lastOtp = await db.emailOtp.findUnique({ where: { email } })
if (lastOtp && Date.now() - lastOtp.lastSentAt.getTime() < 60000) {
  return apiError('Please wait before requesting another OTP', 429)
}
```
